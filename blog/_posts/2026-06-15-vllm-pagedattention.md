## 기존 시스템은 LLM을 어떻게 추론하는가

논문을 제대로 이해하려면 LLM이 어떻게 추론하는지부터 봐야 한다. 추론 과정은 크게 두 단계로 나뉜다.

![LLM 추론 흐름 — prefill과 decode 단계](/assets/img/vllm-inference-overview.png)

먼저 **프롬프트 단계(prefill)** 다. 사용자가 넣은 프롬프트를 통째로 한 번에 모델에 넣어서, 첫 출력 토큰과 프롬프트에 해당하는 모든 토큰의 K와 V를 계산한다. 입력이 한꺼번에 주어지니까 행렬-행렬 곱으로 병렬 처리가 되고, 그래서 GPU를 잘 쓴다.

다음은 **생성 단계(decode)** 다. 여기서부터는 토큰을 하나씩 순서대로 뽑는다(autoregressive). 새로 나온 토큰 하나의 K와 V만 계산하고, 앞 토큰들의 K와 V는 다시 만들지 않고 저장해 둔 값을 그대로 쓴다. 이렇게 저장해 두는 값이 바로 **KV 캐시**다. 캐시가 없으면 토큰을 하나 만들 때마다 지금까지 나온 토큰을 전부 다시 계산해야 해서, KV 캐시는 사실상 없으면 안 되는 장치다. 대신 시퀀스가 길어질수록 캐시도 같이 커진다.

처리량을 올리려면 여러 요청을 한 배치로 묶어야 한다. 그런데 요청마다 길이가 제각각이라, 요즘 시스템(Orca 같은)은 한 스텝이 끝날 때마다 끝난 요청은 빼고 새 요청을 채워 넣는 iteration-level 스케줄링(continuous batching)을 쓴다. 이렇게 하면 GPU가 노는 시간을 줄일 수 있다.

*출처: [Inside Real-Time LLM Inference: From Prefill to Decode, Explained](https://medium.com/@devsp0703/inside-real-time-llm-inference-from-prefill-to-decode-explained-72a1c9b1d85a) — Dev Patel, Medium*

## 왜 메모리가 병목일까

문제는 이 생성 단계다. 토큰을 하나씩 뽑는 decode는 행렬-행렬 곱이 아니라 행렬-벡터 곱이어서 GPU의 연산 능력을 거의 못 쓴다. 그러니까 LLM 생성은 연산보다 메모리에 묶여 있는(memory-bound) 작업이다.

이걸 메우는 방법이 앞에서 본 배치다. 여러 요청을 한 번에 묶으면 똑같은 가중치로 더 많은 일을 시킬 수 있어서 GPU 활용률이 올라간다. 결국 처리량은 한 번에 배치로 묶을 수 있는 요청 수로 갈리고, 그 한계를 긋는 것이 GPU 메모리다.

![Figure 1: 13B 모델 서빙 시 메모리 분포](/assets/img/vllm-fig1-memory-layout.png)
*Figure 1. 13B 모델을 A100 40GB에서 서빙할 때 메모리 분포. 파라미터(65%)는 고정이고, KV 캐시(30%+)만 요청마다 동적으로 늘었다 줄어든다.*

GPU 메모리는 크게 둘로 나뉜다. 모델 가중치는 서빙하는 내내 고정이고(13B 모델이면 약 26GB, 전체의 65% 정도), 나머지에서 30% 넘게 KV 캐시가 차지한다. KV 캐시는 앞에서 말한, 이미 만들어진 토큰들의 Key와 Value를 저장해 두는 공간이다.

문제는 기존 시스템이 이 KV 캐시를 담는 방식이었다. 딥러닝 프레임워크는 대부분 텐서를 연속된 메모리에 두라고 요구한다. 그래서 기존 LLM 서빙 시스템(FasterTransformer, Orca)도 KV 캐시를 연속된 메모리 한 덩어리로 잡았다. 그런데 출력이 몇 토큰에서 끝날지 미리 알 수가 없으니까, 일단 모델이 낼 수 있는 최대 길이(예를 들어 2048 토큰)만큼 공간을 통째로 예약해 둔다.

이러면 짧게 끝난 요청도 최대 길이만큼 메모리를 붙잡고 있고, 요청마다 크기가 달라서 덩어리 사이에 빈틈도 생긴다. 잡아둔 메모리의 상당 부분이 실제 토큰을 담는 데는 안 쓰인다는 뜻이다. 쓸 수 있는 메모리가 줄면 배치에 넣을 요청 수가 줄고, 그러면 곧장 처리량이 떨어진다. (구체적으로 어디서 새는지는 다음에서 본다.)

게다가 이 문제는 갈수록 더 커진다. GPU 세대가 올라가면서 연산 성능과 메모리 용량이 같이 늘긴 했는데, 느는 속도가 다르다. A100에서 H100으로 가면서 FLOPS는 2배 넘게 뛰었는데 GPU 메모리는 80GB로 거의 그대로다. 연산력은 빠르게 느는데 메모리 용량이 그 속도를 못 따라가는 거다. 메모리를 낭비하는 구조를 그대로 두면, 세대가 올라가서 연산력이 좋아져도 그 성능을 다 못 쓴다. 근본 문제가 안 풀린 채로 남으니까, 병목은 점점 더 메모리 쪽으로 쏠린다.

## KV 캐시는 어디서 낭비되는가?

KV 캐시가 얼마나 큰지부터 보자. OPT-13B에서는 토큰 하나의 KV 캐시가 800KB나 된다.

$$
\underbrace{2}_{K,\,V} \times \underbrace{5120}_{\text{hidden}} \times \underbrace{40}_{\text{layers}} \times \underbrace{2}_{\text{fp16 bytes}} = 800\text{KB / token}
$$

2048 토큰짜리 요청 하나가 1.6GB를 먹는다는 얘기다. 그런데 기존 시스템은 이 공간을 세 가지 방식으로 흘린다.

![Figure 3: 기존 시스템의 세 가지 메모리 낭비](/assets/img/vllm-fig3-fragmentation.png)

*Figure 3. 예약, 내부 단편화, 외부 단편화. 실제 토큰이 차지하는 칸은 일부뿐이고 나머지는 비어 있거나 못 쓴다.*

- **내부 단편화 (internal fragmentation)**: 출력이 얼마나 길어질지 모르니까 일단 최대 길이(예: 2048 토큰)만큼 통째로 잡아둔다. 실제로는 50 토큰에서 끝나도, 나머지 1998 토큰 분량은 그 요청이 끝날 때까지 다른 요청이 못 쓴다.
- **예약 (reservation)**: 곧 생성될 토큰을 위해 미리 잡아둔 공간이다. 결국엔 쓰이지만, 쓰이기 전까지는 그냥 묶여 있다.
- **외부 단편화 (external fragmentation)**: 요청마다 크기가 다른 연속 블록을 할당하다 보면 블록 사이에 못 쓰는 빈틈이 생긴다. buddy allocator 같은 할당자에서 흔히 생기는 문제다.

![Figure 2: 시스템별 KV 캐시 실효 사용률](/assets/img/vllm-fig2-memory-waste.png)

*Figure 2. Orca 계열은 실효 사용률이 20.4~38.2%에 그치지만, vLLM은 96.3%까지 끌어올린다.*

논문에서 잰 실효 사용률이 **20.4~38.2%** 밖에 안 됐다. 잡아둔 KV 캐시 메모리의 60~80%가 정작 토큰을 저장하는 데는 안 쓰이고 있었다는 거다.

낭비가 하나 더 있다. 기존 시스템은 메모리를 **공유하지 못한다.** 한 프롬프트로 여러 답을 뽑거나(parallel sampling) beam search를 돌리면 여러 시퀀스가 같은 프롬프트 KV를 공유할 수 있는데, 연속 메모리 구조에서는 각자 따로 복사해서 들고 있어야 했다.

## 해결 방법

저자들의 관찰은 의외로 단순하다. **운영체제가 똑같은 문제를 이미 풀어놨다는 거다.** 프로세스마다 메모리를 연속으로 주면 단편화가 생기니까, OS는 가상 메모리와 페이징으로 해결했다. 프로세스가 보는 주소는 연속이지만 실제 물리 메모리는 흩어진 페이지로 채우고, 페이지 테이블이 그 매핑을 관리한다.

**PagedAttention**은 이 구조를 KV 캐시에 그대로 옮긴 거다.

| OS | PagedAttention |
| --- | --- |
| 프로세스 | LLM 요청 |
| 가상 페이지 | 논리 KV 블록 |
| 물리 페이지 | 물리 KV 블록 (GPU DRAM) |
| 페이지 테이블 | 블록 테이블 |
| 바이트 | 토큰 |

![Figure 6: 블록 테이블을 통한 논리-물리 블록 매핑](/assets/img/vllm-fig6-block-table.png)

*Figure 6. 논리 KV 블록이 블록 테이블을 거쳐 흩어진 물리 블록으로 매핑된다. 물리적으로 연속일 필요가 없다.*

KV 블록 하나에는 정해진 개수의 토큰(기본값 16개)의 K와 V가 들어간다. 핵심은 **물리 블록이 연속일 필요가 없다**는 점이다. 논리 블록 0, 1, 2가 물리 블록 7, 1, 3에 흩어져 있어도, 블록 테이블이 그 매핑만 들고 있으면 된다.

이렇게 바꾸면 앞에서 본 세 가지 낭비가 거의 사라진다. 블록은 꽉 찼을 때만 새로 할당하니까 미리 잡아두는 예약이 없고, 내부 단편화는 마지막 블록 하나 안에서만 생긴다. 모든 블록 크기가 같으니 외부 단편화는 아예 없다. 그래서 vLLM의 실효 사용률은 **96.3%** 까지 올라간다.

공유도 자연스럽게 풀린다. 여러 논리 블록이 같은 물리 블록을 가리킬 수 있어서, 공유하다가 누가 쓰기를 할 때만 복사하는 **Copy-on-Write**가 된다.

- **Parallel sampling**: 프롬프트 KV는 모든 샘플이 똑같으니까 물리 블록 하나만 두고 reference count만 올린다. 새 토큰을 쓸 때 갈라진다.
- **Beam search**: 후보들끼리 공유하는 prefix가 많아서 절감 효과가 크다.
- **Shared prefix**: 여러 요청이 같은 system prompt를 쓰면 그 KV를 한 번만 저장하고 공유한다.

vLLM은 이 PagedAttention 위에 올린 서빙 엔진이다. 중앙 스케줄러가 블록을 관리하고, 메모리가 부족해지면 시퀀스 전체를 내보냈다가(CPU로 swap하거나 나중에 다시 계산) 자리가 나면 다시 들인다.

## 얼마나 빨라졌나

OPT-13B, 66B, 175B를 ShareGPT(긴 대화)와 Alpaca(짧은 지시문) 트레이스로 평가했다.

![Figure 12: 요청률에 따른 정규화 지연 비교](/assets/img/vllm-fig12-throughput.png)

*Figure 12. 요청률(req/s)이 올라갈 때의 정규화 지연. vLLM은 더 높은 요청률까지 낮은 지연을 유지하다가 뒤늦게 꺾인다.*

| 비교 대상 | vLLM 처리량 |
| --- | --- |
| Orca (Oracle, 출력 길이를 안다고 가정한 이상치) | 1.7~2.7× |
| Orca (Max, 항상 최대 길이 예약) | 2.7~8× |
| FasterTransformer (175B) | 최대 22× |

메모리 공유 효과는 beam search에서 특히 두드러진다. parallel sampling은 6.1~9.8%, beam search는 **37.6~55.2%** 나 메모리를 아꼈다.

다만 트레이드오프가 하나 있다. PagedAttention의 attention 커널 자체는 블록 테이블을 조회하고 비연속 메모리에 접근하느라 FasterTransformer보다 **20~26% 느리다.** 그런데도 전체 처리량은 훨씬 앞선다. 커널이 조금 느려진 대신 배치에 훨씬 많은 요청을 넣을 수 있고, 그게 GPU 활용률을 끌어올리기 때문이다. 이 부분이 개인적으로 제일 인상 깊었다.

블록 크기는 16이 기본값이다. 너무 작으면 GPU 병렬성을 못 살리고, 너무 크면 내부 단편화랑 공유 실패가 늘어서, 실험으로 정한 절충점이라고 한다.

## 엔지니어링 관점에서

### 구현

vLLM은 지금 LLM 서빙에서 거의 표준처럼 쓰인다. 다른 것을 고르려면 vLLM을 쓰지 않을 이유를 댈 수 있어야 할 정도다.

prefix caching은 RAG에서 특히 쓸모가 있을 것 같다. 검색 결과를 프롬프트 앞에 붙이는 구조라면, 같은 문서가 들어가는 요청들이 KV 블록을 공유해서 비용을 줄일 수 있다.

block_size는 워크로드에 맞춰서 봐야 하는 값이다. 짧은 요청이 많으면 16보다 작게, 긴 컨텍스트가 많으면 16에서 32 사이를 직접 재보는 게 맞다.

### 평가

논문이 쓴 정규화 지연(normalized latency = 전체 지연 / 출력 길이)은 처리량이랑 지연을 같이 볼 수 있어서 괜찮은 지표 같다. 단순 req/s만 보면 짧은 요청에 유리하게 왜곡되니까.

ShareGPT와 Alpaca 결과가 꽤 다르다는 점도 눈여겨봤다. 짧은 시퀀스에 큰 모델(175B에 Alpaca)이면 병목이 메모리가 아니라 연산으로 넘어가서(compute-bound) vLLM 이점이 줄어든다. 그러니 내 서비스의 시퀀스 길이 분포를 먼저 알아야 기대치를 잡을 수 있다.

### 서빙과 운영

메모리가 부족할 때 swap이랑 recompute 중에 뭐가 빠른지는 모델 크기랑 CPU와 GPU 사이 대역폭에 달렸다. 큰 모델에서는 swap이 느리고, 짧은 컨텍스트면 다시 계산하는 게 의외로 빠르다(디코딩으로 나온 토큰을 프롬프트에 붙여서 prefill로 한 번에 다시 만들 수 있어서).

분산 설정에서는 KV 캐시 매니저가 중앙 스케줄러에 있고 Worker는 블록 ID만 받는다. 각 Worker가 자기 attention head 몫만 저장하면 되니까 텐서 병렬로 확장하기가 깔끔하다.

## 다른 서빙 엔진과 비교

여기부터는 논문 범위 밖이다. vLLM 논문은 2023년 거라서 아래 엔진들은 다루지 않는다. PagedAttention이 길을 연 다음에 서빙 엔진 생태계가 어떻게 갈라졌는지 따로 정리해 둔다. (기능 차이는 시점에 따라 계속 좁혀지는 중이다.)

| 엔진 | KV 캐시 관리 | 강점 | 약점 |
| --- | --- | --- | --- |
| **vLLM** | PagedAttention | 설치와 운영이 쉽고 모니터링 내장, 모델 지원 폭 | 순수 커널 성능은 TensorRT-LLM에 밀림 |
| **TensorRT-LLM** | paged KV + 커널 컴파일 | NVIDIA GPU에서 최대 성능과 낮은 지연 | 엔진 빌드 번거로움, NVIDIA 전용 |
| **SGLang** | RadixAttention | prefix 재사용과 구조화 생성 | 생태계 성숙도는 vLLM보다 아래 |

**vLLM** — 제일 큰 장점은 솔직히 그냥 쓰기 편하다는 거다. `pip install`로 끝나고, 바로 OpenAI 호환 서버가 뜨고, HuggingFace 모델도 대부분 그대로 돌아간다. `/metrics` Prometheus 엔드포인트가 들어 있어서 throughput, 지연, KV 캐시 사용률 같은 지표를 바로 대시보드에 붙일 수 있다. 운영에 손이 적게 간다. 단점이라면, 커널을 직접 컴파일하지는 않으니까 같은 하드웨어에서 끝까지 짜낼 수 있는 마지막 성능은 TensorRT-LLM에 양보한다는 정도다.

**TensorRT-LLM** — NVIDIA가 자기 GPU에 맞춰서 커널을 컴파일해 한계 성능이랑 낮은 지연을 뽑는 쪽이다. FP8 같은 하드웨어 기능까지 깊게 쓴다. 대신 모델마다 엔진을 미리 빌드해야 하고 설정이 까다롭다. NVIDIA 전용이라 하드웨어 선택지가 묶이고, 새 모델이 나왔을 때 바로 띄우기도 좀 어려운 편이다. paged KV cache 같은 vLLM 아이디어는 나중에 가져갔다.

**SGLang** — KV 공유를 한 발 더 밀어붙인 쪽이다. RadixAttention이라고, 프롬프트들의 공통 prefix를 radix tree로 관리해서 자동으로 재사용한다. 멀티턴 대화나 few-shot, 에이전트처럼 호출이 복잡하게 얽히는 워크로드에서 강하고, 구조화된 출력 제어도 잘 된다. 단점은 vLLM에 비하면 커뮤니티랑 생태계가 아직 작다는 거. 단발성 단순 서빙에서는 vLLM이랑 큰 차이를 못 느낄 수도 있다.

세 엔진의 공통점이 더 중요하다. 셋 다 KV 캐시를 어떻게 잘게 쪼개고 공유하느냐를 두고 경쟁한다. 결국 PagedAttention이 깔아놓은 판 위에서 경쟁하는 셈이다.

## 읽고 나서

읽으면서 가장 먼저 든 생각은 기존 시스템이 이 정도로 비효율적이었나 하는 것이었다. LLM 서빙이 GPU를 알뜰하게 쓰는 줄 알았는데, 실제로는 메모리의 60~80%를 흘려보내고 있었다. 잘 돌아가는 것처럼 보이던 시스템 안쪽이 생각보다 헐거웠다.

이런 문제는 보통 더 정교한 새 알고리즘으로 해결한다고 생각하기 쉬운데, vLLM은 정반대였다. 운영체제에서 쓰는 가상 메모리와 페이징이라는 오래된 개념을 거의 그대로 가져와 풀었다. 새로운 기술이 아니라 수십 년간 검증된 아이디어를 같은 문제에 옮겨온 것인데, 그 발상이 오히려 더 인상 깊었다. 좋은 해법이 꼭 새로워야 하는 것은 아니라는 점을 다시 생각하게 됐다.

단순히 GPU 추론 속도만 끌어올린 것이 아니라는 점도 눈에 띈다. 메모리가 부족할 때 어떤 요청을 내보내고 언제 다시 들일지, swap과 recompute 중 무엇을 고를지까지, 스케줄링과 메모리 관리를 함께 설계했다. 커널 하나를 빠르게 만드는 데서 그치지 않고 시스템 전체를 어떻게 운영할지 고민한 흔적이 곳곳에 보였다.

그동안 vLLM은 많이 쓰인다는 이유로 별 고민 없이 가져다 썼다. 빠르다고 하니 그런가 보다 했다. 이번에 논문을 읽고 나서야 그 속도가 어디서 나오는지, PagedAttention이 정확히 무엇을 하는지 분명해졌다. 같은 도구라도 원리를 알고 쓰는 것과 그렇지 않은 것은 다른 일이라는 점을 느꼈다.

## 더 읽을거리 — 서빙 최적화의 다른 축들

PagedAttention은 메모리 관리 쪽 해법이다. LLM 서빙을 빠르게 만드는 시도는 여러 층에서 동시에 일어난다.

- **FlashAttention (2022)** — attention을 메모리 I/O 관점에서 최적화한 연구다. vLLM의 커널 오버헤드 문제랑 바로 맞닿아 있다.
- **Speculative Decoding (2022)** — 작은 draft 모델로 토큰을 미리 여러 개 뽑고 큰 모델이 한 번에 검증하는 방식이다. 디코딩 횟수 자체를 줄이는 접근.
- **Orca (2022)** — iteration-level scheduling을 처음 제시한 논문이다. vLLM이 이 위에 메모리 관리를 얹은 구조다.
- **GPTQ / AWQ** — 가중치를 양자화해서 모델 자체의 메모리랑 연산을 줄이는 또 다른 층이다.
